# Majestic Dashboard → продукт: архитектура и план миграции

## 1. Что есть сейчас

Один файл `majestic_market_dashboard.html` (~2000 строк): чистый **HTML + CSS + ванильный JavaScript**, без фреймворков и без сборки.

- **Данные пользователя** (журнал сделок, цели, настройки) — в `localStorage` браузера.
- **Рыночные данные** — из публичного API Majestic. Браузер напрямую туда не пускает (CORS), поэтому обходной путь: закладка/консоль/импорт JSON-файла.
- **Данные с вики** (пропуск, гос-цены, работы) — вшиты статическим снимком.
- **Автосейв** — File System Access API (только `localhost`), запуск через `start.command`.
- Нет бэкенда, аккаунтов, БД, логирования.

Для личного инструмента этого достаточно. Для публичного продукта с приложениями — нужен фундамент: бэкенд, аккаунты, единый источник данных.

## 2. Главный сдвиг — появляется бэкенд

Это важнее выбора фреймворка. Серверный «поллер» раз в ~30 минут (как обновляется Majestic) забирает данные всех серверов, кладёт в свою БД и раздаёт клиентам через собственный API.

Что это решает:

- **CORS** — клиенты ходят только к твоему API, не к Majestic. Закладка/консоль/импорт больше не нужны.
- **Лимит 5/60с** — соблюдается централизованно: один поллер на всех пользователей, а не каждый долбит API.
- **Свежесть** — данные живые и мгновенные, без ручных действий.
- **Данные пользователя** — переезжают из `localStorage` в аккаунт + БД: синхрон между вебом и телефоном, конец проблемам с origin и бэкапами.

## 3. Рекомендуемый стек — TypeScript везде

| Слой | Технология | Зачем |
|---|---|---|
| Веб | **Next.js (React) + TypeScript** | Идеален для дашборда с таблицами, быстрый, SEO |
| Приложения | **React Native через Expo** | iOS + Android из одной кодовой базы |
| Общая логика | **Пакет в монорепо** (pnpm + Turborepo) | Расчёты, типы, API-клиент — один раз, для веба и мобилок |
| Бэкенд + БД | **Supabase** (Postgres, Auth, RLS, Realtime) | Аккаунты, данные пользователей, права доступа |
| Поллер данных | **Node-воркер** (Railway / Fly.io) или Supabase cron | Опрос Majestic API → кэш в БД |
| Ошибки | **Sentry** | Краши и исключения на вебе и в приложениях |
| Аналитика/логи | **PostHog** | Продуктовые события, воронки, логи |
| Серверные логи | **pino** | Структурные логи поллера/API |
| Хостинг веба | **Vercel** | Деплой Next.js в один клик |

### Почему TypeScript, а не Flutter

Flutter отличный для мобилок, но это **Dart — другой язык**. Тогда веб остаётся на JS, а всю бизнес-логику (которая у тебя уже написана на JS) придётся **переписывать на Dart** и вести две кодовые базы на двух языках.

С React/TS ты:
- переиспользуешь существующий JS-код расчётов почти как есть;
- пишешь логику один раз и делишь между вебом и приложениями;
- нанимаешь под один язык.

Flutter стоит брать, только если веб для тебя вторичен, а мобилки — главное. Здесь дашборд табличный и веб важен, поэтому TS-стек выигрывает.

## 4. Данные и API

**Поток:** Majestic public API → Поллер (каждые ~30 мин) → таблицы снимков в Postgres → твой API → клиенты (веб + приложения).

- Храним снимки по серверам с таймстампом — сразу получаем историю цен и «движения» без костылей.
- Данные вики (пропуск, гос-цены, работы, рецепты) — тоже в БД, обновляются отдельным скриптом/по расписанию, а не вшиты в клиент.
- Клиент никогда не ходит в Majestic напрямую.

## 5. Аккаунты и миграция данных

- Аккаунты через Supabase Auth (email / OAuth).
- Журнал сделок, цели, себестоимость, настройки → таблицы Postgres с RLS (каждый видит только своё).
- **Импорт из текущей версии:** кнопка «загрузить бэкап» принимает старый `majestic_backup.json` / `majestic_data.json` и переносит журнал в аккаунт. Так ты не потеряешь свои данные при переезде.

## 6. Логирование и наблюдаемость

- **Sentry** — ошибки клиентов и сервера, релизы, source maps.
- **PostHog** — какие вкладки открывают, что считают, воронки, retention; сюда же продуктовые логи-события.
- **pino** — структурные логи поллера и API (успех/ошибки опроса, тайминги).
- Алерты: падение поллера, рост ошибок, аномалии по данным.

## 7. Хостинг и деплой

- **Веб:** Vercel (Next.js).
- **БД/аккаунты:** Supabase (managed Postgres).
- **Поллер:** Railway или Fly.io (маленький постоянный сервис).
- **Приложения:** сборка через Expo EAS → App Store и Google Play.

## 8. План миграции по этапам

**Этап 0 — подготовка (монорепо).** Заводим репозиторий, pnpm + Turborepo, пакеты: `core` (логика), `web`, `mobile`, `api`.

**Этап 1 — бэкенд и данные.** Supabase-проект, схема БД, поллер Majestic → кэш. Свой API отдаёт рыночные данные. Веб-прототип читает из API (без закладок).

**Этап 2 — веб на Next.js.** Переносим вкладки по одной (журнал, мастерская, кухня, пропуск, гос-цены…), логику берём из `core`. Аккаунты, миграция журнала из localStorage.

**Этап 3 — прод-веб.** Домен, Vercel, Sentry + PostHog, реклама (Яндекс), SEO. Первый публичный релиз.

**Этап 4 — приложения.** Expo/React Native поверх того же `core` и API. Экраны под мобилку, push-уведомления (напр. «избранное дошло до цены»). Релиз в App Store / Google Play.

## 9. Риски

- **Правила Majestic.** Публичный монетизируемый (реклама) продукт на их данных и названии — требует их согласия. Ранее партнёрство отклонили и разрешили только название + публичный API. **До прода нужно письменно уточнить** (см. отдельное письмо). Это возможный стоп-фактор — выяснить до, а не после.
- **Нагрузка на API и ToS.** Централизованный поллер должен уважать лимиты и не выглядеть как злоупотребление. Кэш + один запрос на всех — как раз про это.
- **Актуальность данных вики** (пропуск, цены) — обновлять по расписанию, а не вручную.

## 10. Грубая оценка усилий

- Этап 1 (бэкенд + поллер + API): ~1–2 недели.
- Этап 2–3 (веб на Next.js + аккаунты + прод): ~3–5 недель.
- Этап 4 (приложения): ~3–4 недели.

Соло и с переиспользованием текущей логики — реалистично выкатить прод-веб за ~1–1.5 месяца, приложения — ещё через месяц.
